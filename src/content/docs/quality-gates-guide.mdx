## Overview

The Enhanced Quality Gates system provides comprehensive validation before any code reaches Cloudflare Workers. It follows a 6-phase execution order designed to catch issues early and prevent deployment failures.

## Quick Start

### Basic Usage

```bash
# Run quality gates manually
npm run quality-gates

# Run quality gates before release (recommended)
npm run pre-release

# Full release with quality gates
npm run release
```

### What Happens

1. **Quality Gates Run** - 6 phases of validation
2. **Version Bump** - Automatic semantic versioning
3. **Deployment** - To Cloudflare Workers (only if gates pass)

## Phase-by-Phase Breakdown

### Phase 1: Pre-Validation (Fast Failures First)
**Purpose**: Catch environment and configuration issues immediately

- Environment Variables Check (Cloudflare Workers secrets via `wrangler secret list`)
- Configuration Files Validation  
- Dependencies Installation Check

**Why First**: These are the fastest checks that can fail immediately. No point running expensive tests if your environment is broken.

**Note**: Environment variables are configured in Cloudflare Workers settings, not locally required. See [Environment Variables Guide](deployment/environment-variables.mdx) for details.

### Phase 2: Code Quality (Static Analysis)
**Purpose**: Validate code structure and security patterns

- Linting (ESLint)
- Security Pattern Validation
- Type Checking (skipped - happens during build)

**Why This Order**: Linting catches obvious syntax errors first, then we check for security anti-patterns.

### Phase 3: Security & Compliance
**Purpose**: Ensure code security and compliance

- Security Audit (npm audit)
- Vulnerability Scan
- Security Headers Validation

**Why This Order**: Security issues can invalidate everything else, so we check them before running tests.

### Phase 4: Testing & Coverage
**Purpose**: Validate code functionality and test coverage

- Unit Tests (Fast)
- Integration Tests (skipped - not configured)
- E2E Tests (Playwright)
- Coverage Analysis (80% threshold)

**Why This Order**: Unit tests are fastest and catch most issues. E2E tests are slowest but catch integration problems.

### Phase 5: Infrastructure Validation
**Purpose**: Ensure infrastructure compatibility

- Database Schema Validation (Remote D1)
- API Endpoint Registration Check
- Cloudflare Workers Compatibility
- Bundle Size Validation (skipped - requires build)

**Why This Order**: Infrastructure checks depend on tests passing, and Workers compatibility depends on code being valid.

### Phase 6: Performance & Final Checks
**Purpose**: Final validation and performance metrics

- Performance Checks (skipped - requires running app)
- Bundle Analysis (skipped - requires build)
- Final Security Scan
- Deployment Readiness Check

**Why This Order**: Performance checks depend on everything else working, and deployment readiness ensures all gates passed.

## Environment Variables Required

The quality gates require these environment variables to be set:

```bash
# JWT Authentication
JWT_SECRET=your-super-secure-jwt-secret-key-at-least-32-characters-long

# Mailjet Email Service
MAILJET_API_KEY=your_mailjet_api_key_here
MAILJET_API_SECRET=your_mailjet_api_secret_here
MAILJET_FROM_EMAIL=noreply@yourdomain.com
```

## Database Validation

### What It Checks

The quality gates validate your **remote D1 database** against your code expectations:

1. **Table Existence** - Required tables must exist
2. **Column Existence** - Required columns must be present
3. **Data Type Compatibility** - Column types must match expectations
4. **Constraints & Indexes** - Required constraints must exist

### Why Remote Database

Since your code runs against the **remote D1 database** (not local), this validation ensures:
- Your Workers code will actually work
- No missing table/column errors in production
- Schema changes are properly migrated

## Troubleshooting

### Common Issues

#### 1. Environment Variables Missing
```bash
 Missing required environment variables: JWT_SECRET, MAILJET_API_KEY
```
**Solution**: Set required environment variables (see `env.example`)

#### 2. ESLint Errors
```bash
 ESLint failed - fix linting errors before proceeding
```
**Solution**: Run `npm run lint:fix` to auto-fix issues

#### 3. Test Failures
```bash
 Unit tests failed - fix test failures before proceeding
```
**Solution**: Fix failing tests or update test expectations

#### 4. Database Schema Mismatch
```bash
 Database validation failed: Missing required table: library_users
```
**Solution**: Run database migrations or update schema

#### 5. Security Vulnerabilities
```bash
 Security audit failed - fix vulnerabilities before proceeding
```
**Solution**: Run `npm audit fix` or manually resolve vulnerabilities

### Skipped Phases

Some phases are intentionally skipped in development:
- **Bundle Size Validation** - Requires build step
- **Performance Checks** - Requires running application
- **Integration Tests** - Not configured yet

These can be enabled as needed.
