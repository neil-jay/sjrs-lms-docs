## Overview

This document defines the retention policy for system logs stored in the `system_logs` table. Proper log retention is critical for compliance, performance, and storage management.

## Retention Periods

### Production Logs

| Log Level | Retention Period | Rationale |
|-----------|------------------|-----------|
| **CRITICAL/ERROR** | 365 days (1 year) | Required for incident investigation, compliance audits, and root cause analysis |
| **WARN** | 180 days (6 months) | Useful for trend analysis and identifying recurring issues |
| **INFO** | 90 days (3 months) | General operational visibility and debugging |
| **DEBUG** | 30 days (1 month) | Development/troubleshooting only, high volume |

### Special Categories

- **Security-related logs** (authentication failures, permission denials, suspicious activity): **365 days minimum**
- **Audit logs** (separate from system logs): **7 years** (compliance requirement)

## Implementation

### Automated Cleanup (Recommended)

System logs should be automatically purged based on the retention policy using Cloudflare Workers Cron Triggers.

**Location**: `functions/scheduled/cleanup-system-logs.ts`

```typescript
// Scheduled daily at 2:00 AM UTC
export async function cleanupSystemLogs(env: Environment): Promise<void> {
  const now = new Date();
  
  // Delete logs based on level-specific retention
  await env.DB.prepare(`
    DELETE FROM system_logs 
    WHERE 
      (level = 'DEBUG' AND created_at < datetime('now', '-30 days'))
      OR (level = 'INFO' AND created_at < datetime('now', '-90 days'))
      OR (level = 'WARN' AND created_at < datetime('now', '-180 days'))
      OR (level IN ('ERROR', 'CRITICAL') AND created_at < datetime('now', '-365 days'))
  `).run();
}
```

### Manual Cleanup (Emergency)

For manual cleanup or adjustments:

```sql
-- Review log volume by level
SELECT level, COUNT(*) as count, 
       MIN(created_at) as oldest, 
       MAX(created_at) as newest
FROM system_logs 
GROUP BY level;

-- Delete old DEBUG logs (older than 30 days)
DELETE FROM system_logs 
WHERE level = 'DEBUG' 
AND created_at < datetime('now', '-30 days');

-- Delete old INFO logs (older than 90 days)
DELETE FROM system_logs 
WHERE level = 'INFO' 
AND created_at < datetime('now', '-90 days');

-- Delete old WARN logs (older than 180 days)
DELETE FROM system_logs 
WHERE level = 'WARN' 
AND created_at < datetime('now', '-180 days');

-- Delete old ERROR/CRITICAL logs (older than 365 days)
DELETE FROM system_logs 
WHERE level IN ('ERROR', 'CRITICAL') 
AND created_at < datetime('now', '-365 days');
```

## Monitoring

### Log Volume Alerts

Set up alerts when log volume exceeds thresholds:

- **Total logs > 1,000,000**: Review retention policy
- **Daily ERROR logs > 1,000**: Investigate systemic issues
- **Daily CRITICAL logs > 10**: Immediate investigation required

### Storage Monitoring

Monitor D1 database size and log table growth:

```sql
-- Estimate table size (approximate)
SELECT 
  COUNT(*) as total_logs,
  SUM(LENGTH(message) + LENGTH(context) + LENGTH(COALESCE(error_details, ''))) as approx_size_bytes
FROM system_logs;
```

## Archive Strategy (Optional)

For long-term storage beyond retention periods:

1. **Export to Object Storage**: Use Cloudflare R2 or similar
2. **Compressed Format**: Export as `.csv.gz` or `.parquet`
3. **Archive Retention**: 7 years for compliance
4. **Access**: Cold storage, retrieve only for audits/investigations

### Archive Export Script

```bash
# Export logs to CSV before deletion
npm run export:logs -- --start-date 2025-01-01 --end-date 2025-12-31 --output archive/logs-2025.csv
```

## Compliance Considerations

### Data Privacy

- **PII Sanitization**: Ensure sensitive data is not logged in system logs
- **GDPR**: System logs containing user identifiers must respect data subject rights
- **Right to Erasure**: Implement user data removal from logs when requested

### Audit Trail

- Log all retention policy changes in audit logs
- Record deletion operations (date, count, criteria)
- Maintain evidence of policy compliance

## Performance Impact

### Database Performance

- **Index Optimization**: Ensure `created_at` column is indexed for efficient deletion
- **Batch Deletion**: Delete logs in batches to avoid locking
- **Off-Peak Execution**: Run cleanup during low-traffic hours (2-4 AM UTC)

### Query Performance

Regular cleanup maintains optimal query performance:

- Reduces table scan time
- Keeps statistics current
- Prevents index bloat

## Review Schedule

This retention policy should be reviewed:

- **Quarterly**: Verify compliance and adjust if needed
- **After Incidents**: If logs were insufficient for investigation
- **Annually**: Full policy review and adjustment
