## 🔧 Development Workflow

### Code Quality & Automation

#### Automated Quality Checks
- **Pre-commit Hooks** (`.husky/pre-commit`):
  - ✅ ESLint auto-fix on staged files (via `lint-staged`)
  - ✅ File size warnings (non-blocking) for staged files
  - Runs automatically before every commit

- **Pre-push Hooks** (`.husky/pre-push`):
  - ✅ TypeScript type checking (blocks push if errors)
  - ✅ ESLint validation (blocks push if errors)
  - ✅ File size checks (warnings only)
  - Catches issues before code reaches GitHub Actions

- **File Size Standards**:
  - Utilities/Handlers: ≤ 300 lines
  - Pages: ≤ 500 lines
  - Components: ≤ 300 lines
  - Check manually: `npm run check:file-sizes`
  - Check staged files: `npm run check:file-sizes -- --staged`

#### Code Quality Tools
- **TypeScript**: Enhanced configuration with balanced strictness
- **ESLint**: Code linting and style enforcement
- **Prettier**: Code formatting
- **Husky**: Git hooks management
- **lint-staged**: Run linters on staged files only

## 📖 Contributing

### Code Standards

#### File Size Limits
- **Utilities/Handlers**: ≤ 300 lines
- **Pages**: ≤ 500 lines
- **Components**: ≤ 300 lines
- Files exceeding limits are automatically flagged in pre-commit/pre-push hooks
- See [File Size Analysis](../architecture/file-size-analysis.mdx) for details

#### Error Handling
- Use `handleError` from `functions/utilities/logging` for all error logging
- Use `createErrorResponse` from `functions/utilities/response-builder` for error responses
- Use `ERROR_CODES` constants (never string literals) for error codes
- Include component name in error context for better tracking
- See [API Response Standardization](../api/api-response-standardization.mdx) for patterns

#### General Standards
- Follow TypeScript best practices
- Use consistent naming conventions
- Add appropriate error handling
- Include JSDoc comments for complex functions
- Use safe SQL query builders (`buildSafeOrderByClause`, `buildSafeUpdateQuery`)

### Pull Request Process
1. Create feature branch from main
2. Implement changes with tests
3. Ensure TypeScript compilation passes
4. Submit PR with clear description
5. Address review feedback

### Testing Requirements
- All new features must include tests
- Maintain or improve test coverage
- Ensure tests pass in CI/CD pipeline
