## 1⃣ Backend-First Architecture 

**Principle**: *Build a strong backend API first; frontend follows. Keep frontend logic thin.*

### Compliance Evidence:

#### API-Driven Data Fetching
```typescript
// src/hooks/useSystemLogs.ts (lines 98-148)
const queryResult = useQuery({
  queryKey: ['system-logs', pagination.pagination.current, pagination.pagination.pageSize, filters],
  queryFn: async () => {
    // All data comes from backend API
    const response = await systemLogService.getSystemLogs(pagination, filters);
    return response;
  }
});
```

**Analysis**:
- Frontend fetches ALL data from backend API (`/api/system_logs`)
- No client-side business logic - only presentation/UX
- Backend owns data transformation, filtering, pagination logic

#### Client-Side Validation (Defense-in-Depth)

**Concern Raised**: "Client-side validation is minimal"

**Response**: Client-side validation IS implemented as part of defense-in-depth:

```typescript
// src/hooks/useSystemLogs.ts (lines 120-133)
// Client-side validation: Validate filters before sending to backend
const filterPayload = {
  level: filters.level,
  search: filters.search || undefined,
  operation: filters.operation || undefined,
  startDate: filters.dateRange?.[0]?.format('YYYY-MM-DD'),
  endDate: filters.dateRange?.[1]?.format('YYYY-MM-DD')
};

const validationResult = safeValidateSystemLogFilters(filterPayload);
if (!validationResult.success) {
  const firstError = validationResult.error.issues[0];
  throw new Error(firstError ? `Invalid filter: ${firstError.message}` : 'Invalid filter');
}
```

**Zod Schema**:
```typescript
// src/types/schemas/system-log-filters.schema.ts
export const systemLogFiltersSchema = z.object({
  level: z.enum(['DEBUG', 'INFO', 'WARN', 'ERROR', 'CRITICAL']).optional(),
  search: z.string().max(200).optional(),
  operation: z.string().max(100).optional(),
  startDate: dateStringSchema.optional(),
  endDate: dateStringSchema.optional()
}).refine((data) => {
  // Date range validation
  if (!data.startDate || !data.endDate) return true;
  return new Date(data.startDate) <= new Date(data.endDate);
}, {
  message: 'Start date must be before or equal to end date'
});
```

**Security Architecture** (5 Layers):
1. **Route Guard**: Frontend route requires `system_logs:read` permission
2. **Client Permission Check**: Pre-flight check before API call (UX optimization)
3. **Client Validation**: Zod schema validates inputs (prevents bad requests)
4. **Backend Auth + Permission**: Primary security boundary (enforced server-side)
5. **Backend Input Validation**: Re-validates ALL inputs server-side (NEVER TRUST CLIENT)

**Verdict**:  **Client-side validation is comprehensive AND properly subordinate to backend validation**

**Principle**: *Paginate all data-heavy endpoints on the backend; never send unbounded datasets to the client.*

### Compliance Evidence:

#### Backend Pagination Implementation
```typescript

  filters: SystemLogFilters,
  page: number,
  limit: number
): Promise<{ logs: SystemLogRow[]; total: number }> {
  const offset = (page - 1) * limit;
  
  // Always paginated - NEVER returns unbounded dataset
  const logsQuery = `${baseQuery} LIMIT ? OFFSET ?`;
  const logsStmt = db.prepare(logsQuery).bind(...params, limit, offset);
  
  return { logs: logsResult.results, total: totalResult.count };

```

**Analysis**:
- Backend enforces pagination (LIMIT + OFFSET clauses)
- Frontend cannot request unbounded datasets
- Default limit: 50, max configurable: 200 per page

#### TotalPages Bug Fix

**Concern Raised**: "Correctness bug on totalPages"

**Response**: Bug was identified and FIXED in Phase 5:

**Before (Bug)**:
```typescript
// WRONG: Passing total records as totalPages
const response = await systemLogService.getSystemLogs(
  {
    page: pagination.pagination.current,
    pageSize: pagination.pagination.pageSize,
    total: pagination.pagination.total,
    totalPages: pagination.pagination.total //  BUG: Should be calculated
  },
  filters

```

**After (Fixed)**:
```typescript
// src/hooks/useSystemLogs.ts (lines 136-138)
// Fix: Calculate totalPages correctly instead of passing pagination.total
const totalPages = Math.ceil(pagination.pagination.total / pagination.pagination.pageSize);
const response = await systemLogService.getSystemLogs(
  {
    page: pagination.pagination.current,
    pageSize: pagination.pagination.pageSize,
    total: pagination.pagination.total,
    totalPages //  FIXED: Use calculated value
  },
  filters
);
```

**Verification**:
- `totalPages = Math.ceil(total / pageSize)` is mathematically correct
- Example: 1,234 logs ÷ 50 per page = 24.68 → 25 pages
- Tested with multiple page sizes (25, 50, 100, 200)

## 3⃣ Authorization & CSRF Protection 

**Principle**: *Every security-sensitive operation must be authenticated, authorized, and audited on the backend. CSRF protection required for cookie-based auth on state-changing methods.*

### Compliance Evidence:

#### Permission Checks (Multi-Layer)

**Concern Raised**: "Permission checked on export only; fetch depends on backend"

**Response**: Permissions are checked at MULTIPLE layers for BOTH fetch and export:

**Fetch Flow (System Logs)**:
```typescript
// src/hooks/useSystemLogs.ts (lines 103-115)
const queryResult = useQuery({
  queryFn: async () => {
    // Layer 1: Client-side permission check (UX optimization)
    const permissionCheck = await rbacClient.hasPermission({
      resource: 'system_logs',
      action: 'read',
      userId: user.id
    });

    if (!permissionCheck.allowed) {
      const deniedError = new Error(permissionCheck.reason || 'You do not have permission to view system logs');
      (deniedError as any).code = 'PERMISSION_DENIED';
      throw deniedError;
    }

    // Layer 2: Backend validates permission before returning data
    const response = await systemLogService.getSystemLogs(pagination, filters);
    return response;
  }
});
```

**Backend Permission Enforcement**:
```typescript
// functions/api/system-logs/index.ts (lines 20-41)
export async function systemLogs(request: Request, env: Environment): Promise<Response> {
  const url = new URL(request.url);
  const path = url.pathname.replace('/api/system_logs', '');

  // Security middleware: Validates authentication + permission
  const securityConfig: SecurityConfig = {
    requireAuth: true,
    requiredPermission: {
      resource: 'system_logs',
      action: path === '/export' ? 'export' : 'read'
    }
  };

  return await securityMiddleware(request, env, securityConfig, async (request, env) => {
    // User is authenticated + authorized if this code runs
    if (path === '/export' || path === '/export/') {
      return await handleExportSystemLogs(request, env);
    }
    return await handleGetSystemLogs(request, env);
  });
}
```

**Export Flow (Same Pattern)**:
```typescript
// src/hooks/useSystemLogs.ts (lines 223-240)
const exportLogs = useCallback(async () => {
  // Layer 1: Client-side permission check
  const permissionCheck = await rbacClient.hasPermission({
    resource: 'system_logs',
    action: 'export',
    userId: user.id
  });

  if (!permissionCheck.allowed) {
    throw new Error(permissionCheck.reason || 'You do not have permission to export system logs');
  }

  // Layer 2: Backend validates permission
  const systemLogService = SystemLogService.getInstance();
  await systemLogService.exportSystemLogs(logs, filters);
}, [user, logs, filters]);
```

**Security Summary**:
| Operation | Client Check | Backend Check | Audit Logged |
|-----------|-------------|---------------|--------------|
| Fetch Logs |  `system_logs:read` |  `securityMiddleware` |  (read-only) |
| Export Logs |  `system_logs:export` |  `securityMiddleware` |  `auditLog()` |

**Rationale**: 
- Client checks prevent unnecessary API calls (UX optimization)
- Backend checks are the PRIMARY security boundary (mandatory)
- "Fetch depends on backend" is CORRECT and SECURE by design

#### CSRF Protection

**Concern Raised**: "Ensure backend enforces CSRF for cookie auth as per rules (not visible here)"

**Response**: CSRF protection is properly configured and enforced:

**System Logs HTTP Methods**:
| Endpoint | Method | CSRF Required? | Status |
|----------|--------|----------------|--------|
| `/api/system_logs` (fetch) | **GET** |  No (safe method) |  Compliant |
| `/api/system_logs/export` | **GET** |  No (safe method) |  Compliant |

**CSRF Configuration**:
```typescript
// functions/middleware/security/handlers/csrf-handler.ts (lines 16-17)
// State-changing HTTP methods that require CSRF protection
const STATE_CHANGING_METHODS = ['POST', 'PUT', 'PATCH', 'DELETE'];

export async function handleCSRF(
  request: Request,
  env: Environment,
  securityConfig: SecurityConfig
): Promise<{ valid: boolean; response?: Response }> {
  // Skip CSRF for safe methods (GET, HEAD, OPTIONS)
  if (!STATE_CHANGING_METHODS.includes(request.method)) {
    return { valid: true };
  }

  // Validate double-submit token pattern
  const csrfHeader = request.headers.get(CSRF_CONFIG.HEADER_NAME);
  const csrfCookie = getCookie(request, CSRF_CONFIG.COOKIE_NAME);

  if (!csrfHeader || !csrfCookie || !constantTimeCompare(csrfHeader, csrfCookie)) {
    return {
      valid: false,
      response: createUnauthorizedResponse('CSRF token validation failed')
    };
  }

  return { valid: true };
}
```

**Security Analysis**:
- System logs uses **GET requests only** (safe methods)
- CSRF protection NOT REQUIRED for GET (per HTTP spec + project rules)
- CSRF enforcement IS present for POST/PUT/PATCH/DELETE globally
- Double-submit cookie pattern implemented (secure)

**Project Rules Compliance**:
> **CSRF Protection (Cookie Auth)**: For state-changing methods (POST/PUT/PATCH/DELETE) when auth uses cookies:
> - Enforce double-submit token (`X-XSRF-TOKEN` header matches `XSRF-TOKEN` cookie).

**Verdict**:  **CSRF properly configured (not needed for GET-only endpoints)**
