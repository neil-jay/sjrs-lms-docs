This document provides comprehensive documentation for the Badge System in the SJRS LMS platform.

## Overview

The Badge System is a gamification feature that recognizes and rewards user achievements and milestones. Badges can be automatically awarded based on user actions (e.g., first visit, loan milestones, reviews) or manually assigned by administrators. The system supports both explicit badge assignments and role-based virtual badges.

## Key Features

- **Automatic Badge Awarding**: System automatically grants badges based on user actions and milestones
- **Manual Badge Assignment**: Administrators can manually assign badges to users
- **Role-Based Badges**: Virtual badges automatically assigned based on user roles
- **Badge Expiration**: Support for time-limited badge assignments
- **Badge Revocation**: Ability to revoke badges when needed
- **Comprehensive Audit Logging**: All badge operations are logged for security and compliance
- **Security Controls**: Multiple layers of security to prevent unauthorized badge manipulation

## Database Schema

### Core Tables

**`badges` Table**
- `id` - Primary key
- `key` - Unique identifier for the badge (e.g., 'welcome_member', 'loan_10')
- `name` - Display name of the badge
- `description` - Detailed description of the badge
- `icon` - Icon identifier or URL
- `level` - Badge level (1-10, higher levels require superuser privileges for assignment)
- `category` - Badge category for organization
- `is_active` - Whether the badge is currently active
- `created_at`, `updated_at` - Timestamps

**`user_badges` Table**
- `id` - Primary key
- `user_id` - Foreign key to `library_users`
- `badge_id` - Foreign key to `badges`
- `granted_at` - When the badge was awarded
- `expires_at` - Optional expiration date (ISO datetime format)
- `is_revoked` - Whether the badge has been revoked
- `revoked_at` - When the badge was revoked (if applicable)
- `reason` - Optional reason for assignment/revocation
- `granted_by` - User ID who granted the badge (for manual assignments)
- `created_at`, `updated_at` - Timestamps

**`role_badges` Table** (Virtual Badges)
- `id` - Primary key
- `role` - User role (e.g., 'student', 'professor', 'librarian', 'admin', 'superuser')
- `badge_id` - Foreign key to `badges`
- `created_at`, `updated_at` - Timestamps

## Badge Types

### 1. Automatic Badges

These badges are automatically awarded by the system based on user actions:

#### Welcome Badges
- **`welcome_member`**: Awarded when a user first visits the platform
- **`approved_member`**: Awarded when a user's account is approved

#### Visit Badges
- **`visit_1`**: First visit
- **`visit_5`**: 5 visits
- **`visit_10`**: 10 visits
- **`visit_25`**: 25 visits
- **`visit_50`**: 50 visits
- **`visit_100`**: 100 visits
- **`visit_250`**: 250 visits
- **`visit_500`**: 500 visits
- **`visit_1000`**: 1000 visits

#### Loan Badges
- **`loan_1`**: First loan
- **`loan_5`**: 5 loans
- **`loan_10`**: 10 loans
- **`loan_25`**: 25 loans
- **`loan_50`**: 50 loans
- **`loan_100`**: 100 loans
- **`loan_250`**: 250 loans
- **`loan_500`**: 500 loans
- **`loan_1000`**: 1000 loans

#### Review Badges
- **`review_1`**: First review
- **`review_5`**: 5 reviews
- **`review_10`**: 10 reviews
- **`review_25`**: 25 reviews
- **`review_50`**: 50 reviews

### 2. Role-Based Badges (Virtual)

These badges are automatically assigned based on user roles and are not stored in `user_badges`. They appear dynamically in user badge views:

- **Student Badges**: Automatically shown for users with 'student' role
- **Professor Badges**: Automatically shown for users with 'professor' role
- **Librarian Badges**: Automatically shown for users with 'librarian' role
- **Admin Badges**: Automatically shown for users with 'admin' role
- **Superuser Badges**: Automatically shown for users with 'superuser' role

**Important**: Role-based badges cannot be manually assigned or revoked. They are tied to the user's role and appear/disappear automatically when the role changes.

## API Endpoints

### 1. Get All Badges (Catalog)
- **Endpoint**: `GET /api/badges`
- **Permissions**: Requires `badges:read` permission
- **Description**: Retrieves all available badges in the system
- **Response**: List of badge objects with details

### 2. Get My Badges
- **Endpoint**: `GET /api/badges/me`
- **Permissions**: User can view their own badges
- **Description**: Retrieves all badges for the authenticated user (includes both explicit assignments and role-based badges)
- **Response**: List of user's badges with status information

### 3. Get Granted Badges (Admin View)
- **Endpoint**: `GET /api/badges/granted`
- **Permissions**: Requires `badges:read` permission
- **Description**: Retrieves all badge assignments across all users with filtering and pagination
- **Query Parameters**:
  - `page` - Page number (default: 1)
  - `pageSize` - Items per page (default: 20)
  - `status` - Filter by status: `'active'`, `'revoked'`, or `'any'` (default: `'any'`)
  - `user_id` - Filter by specific user ID
  - `badge_key` - Filter by badge key
  - `search` - Search by user name or email
- **Response**: Paginated list of badge assignments with user and badge details
- **Features**:
  - Includes both explicit assignments (`user_badges`) and role-based badges
  - Filters out expired badges when status is `'active'`
  - Supports comprehensive search and filtering

### 4. Assign Badge
- **Endpoint**: `POST /api/badges/assign`
- **Permissions**: Requires `badges:create` permission
- **Description**: Manually assign a badge to a user
- **Request Body**:
  ```json
  {
    "user_id": 123,
    "badge_key": "loan_10",
    "reason": "Milestone achievement",
    "expires_at": "2025-12-31T23:59:59Z" // Optional, ISO datetime format
  }
  ```
- **Response**: Success message with badge assignment details
- **Security Features**:
  - Prevents self-assignment (users cannot assign badges to themselves)
  - Blocks manual assignment of role-based badges
  - Requires superuser privileges for badges with `level >= 5`
  - Validates `expires_at` format and ensures it's in the future
  - Logs all assignments in audit log

### 5. Revoke Badge
- **Endpoint**: `POST /api/badges/revoke`
- **Permissions**: Requires `badges:delete` permission
- **Description**: Revoke a badge from a user
- **Request Body**:
  ```json
  {
    "user_id": 123,
    "badge_key": "loan_10",
    "reason": "Violation of terms"
  }
  ```
- **Response**: Success message with revocation details
- **Security Features**:
  - Prevents revocation of role-based badges
  - Verifies badge is actually assigned before revocation
  - Checks if badge is already revoked
  - Logs all revocations in audit log

## Security Features

### 1. Self-Assignment Prevention
Users cannot assign badges to themselves. This prevents privilege escalation and ensures proper oversight.

### 2. Role-Based Badge Protection
Role-based badges cannot be manually assigned or revoked. They are automatically tied to user roles and managed by the system.

### 3. High-Level Badge Protection
Badges with `level >= 5` can only be assigned by superusers. This prevents unauthorized assignment of high-priority badges.

### 4. Input Validation
- `expires_at` must be a valid ISO datetime string
- `expires_at` must be in the future (not in the past)
- All required fields are validated before processing

### 5. Audit Logging
All badge operations are logged in the `action_logs` table with:
- Action type (`badge_assigned`, `badge_reassigned`, `badge_revoked`)
- User who performed the action
- Target user and badge details
- Timestamp and IP address
- User agent information

### 6. Expired Badge Filtering
When viewing active badges, expired badges are automatically filtered out. The system checks both `is_revoked = 0` and `expires_at > now()` for active status.

## Automatic Badge Evaluation

The system includes evaluators that automatically check and award badges based on user actions:

### Welcome Evaluator (`functions/lib/badges/evaluators/welcome.ts`)
- Checks for first visit and account approval
- Awards `welcome_member` and `approved_member` badges

### Visit Evaluator (`functions/lib/badges/evaluators/visits.ts`)
- Tracks user visit counts
- Awards milestone badges (`visit_1`, `visit_5`, `visit_10`, etc.)

### Loan Evaluator (`functions/lib/badges/evaluators/loans.ts`)
- Tracks loan counts
- Awards milestone badges (`loan_1`, `loan_5`, `loan_10`, etc.)

### Review Evaluator (`functions/lib/badges/evaluators/reviews.ts`)
- Tracks review counts
- Awards milestone badges (`review_1`, `review_5`, `review_10`, etc.)

### Approval Evaluator (`functions/lib/badges/evaluators/approval.ts`)
- Monitors account approval status
- Awards `approved_member` badge

## User Interface

### Badge Management Page (`/dashboard-superuser/badges`)

The badge management interface is organized into three main tabs:

#### 1. Badge Catalog Tab
- Displays all available badges in the system
- Shows badge details: name, description, icon, level, category
- Allows administrators to view badge information

#### 2. Badge Assignment Tab
- Interface for manually assigning badges to users
- Includes user selection, badge selection, reason field, and optional expiration date
- Pre-fills badge selection when navigating from Badge Catalog

#### 3. Granted Badges Tab
- Displays all badge assignments across all users
- Includes filtering by status (active, revoked, any)
- Supports search by user name or email
- Shows both explicit assignments and role-based badges
- Displays expiration dates and revocation status
- Pagination support for large datasets

## Best Practices

### For Administrators
1. **Use Appropriate Badge Levels**: Reserve high-level badges (level 5+) for significant achievements
2. **Set Expiration Dates**: Use expiration dates for time-limited achievements or promotional badges
3. **Document Reasons**: Always provide a reason when manually assigning or revoking badges
4. **Monitor Audit Logs**: Regularly review badge assignment logs for security and compliance
5. **Respect Role-Based Badges**: Understand that role-based badges cannot be manually managed

### For Developers
1. **Automatic Evaluation**: Integrate badge evaluators into relevant user action flows
2. **Error Handling**: Always handle badge award failures gracefully (don't block user actions)
3. **Performance**: Badge evaluation should be asynchronous and non-blocking
4. **Testing**: Test badge evaluation logic thoroughly, especially edge cases
5. **Security**: Never bypass security checks when implementing badge features

## Troubleshooting

### Issue: Badge not showing in "Granted Badges" tab
- **Check**: Ensure the badge is not expired (if status filter is 'active')
- **Check**: Verify the badge is not revoked (if status filter is 'active')
- **Check**: For role-based badges, ensure the user has the correct role

### Issue: Cannot assign badge
- **Check**: Verify you have `badges:create` permission
- **Check**: Ensure you're not trying to assign a role-based badge
- **Check**: For high-level badges (level 5+), verify you're a superuser
- **Check**: Ensure you're not trying to assign a badge to yourself

### Issue: Cannot revoke badge
- **Check**: Verify you have `badges:delete` permission
- **Check**: Ensure the badge is not a role-based badge
- **Check**: Verify the badge is actually assigned to the user

### Issue: Expired badges still showing
- **Check**: Ensure the status filter is set to 'active' (expired badges are filtered out for active status)
- **Check**: Verify the `expires_at` field is correctly formatted (ISO datetime)
