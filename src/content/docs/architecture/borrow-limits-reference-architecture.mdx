### 5. Pagination: Server-Side Implementation ✅

#### Backend Expects: `page` + `limit`
```typescript
// functions/api/borrow-limits/handlers/get-borrow-limits.ts
const rawFilters = {
  page: searchParams.page ? parseInt(searchParams.page) : 1,
  limit: searchParams.limit ? parseInt(searchParams.limit) : 20  // ✅ 'limit'
};
```

#### Frontend Hook Sends: `page` + `limit`
```typescript
// src/hooks/d1/borrowLimits.ts
const params: Record<string, string | number> = {};
if (pagination) {
  params.page = pagination.current;
  params.limit = pagination.pageSize;  // ✅ Matches backend 'limit'
}
```

#### List Page Passes Pagination to Query
```tsx
// src/pages/borrow-limits/BorrowLimitList.tsx
const [pagination, setPagination] = useState({
  current: 1,
  pageSize: 20,
  total: 0,
});

// ✅ Pass pagination to enable server-side pagination
// ✅ Use React Query defaults (no aggressive polling)
const { data: limitsData, isLoading, error } = useD1BorrowLimitsQuery({ 
  pagination,
});
// React Query automatically refetches on:
// - Component mount (refetchOnMount: true)
// - Window focus (refetchOnWindowFocus: true)
// - Manual invalidation after mutations
// No aggressive polling needed - this is optimal for performance

// ✅ Update total from server response
useEffect(() => {
  if (limitsData?.total !== undefined) {
    setPagination(prev => ({
      ...prev,
      total: limitsData.total
    }));
  }
}, [limitsData?.total]);

// ✅ Handle page changes
const handleTableChange = (page: number, pageSize: number) => {
  setPagination((prev) => ({
    ...prev,
    current: page,
    pageSize,
  }));
};
```

**Verification Points**:
- ✅ Backend parameter name matches frontend: `limit` (not `pageSize`)
- ✅ List page passes `pagination` state to query
- ✅ **No aggressive polling**: Uses React Query defaults (refetchOnMount, refetchOnWindowFocus)
- ✅ Mutations trigger cache invalidation (automatic refresh)

**Result**: Proper server-side pagination with optimal performance - no unnecessary polling
- ✅ No unbounded dataset fetching
- ✅ Follows "Backend-First" and "Pagination" principles

**Result**: Proper server-side pagination - no unbounded datasets, backend controls data flow.

## 🎯 What Makes This a Good Example

### 1. **Security First**
- Every page checks permissions before rendering
- Users see clear "Access Denied" messages
- Actions are conditionally available based on permissions
- Backend enforces security; frontend provides UX

### 2. **Clean Architecture**
- No thin wrapper re-exports
- Direct imports from shared code
- Feature module is pure presentation
- Clear separation: data (shared) vs. UI (feature)

### 3. **Import Hygiene**
- No deep relative imports in pages
- Minimal deep imports in feature (1 acceptable case)
- Easy to follow where code comes from
- No circular dependency risk

### 4. **Server-Side Pagination**
- Backend controls data flow (not client)
- List page passes pagination state to query
- Parameter names match backend expectations
- Total count synced from server
- No unbounded dataset fetching
- **No aggressive polling** - uses React Query defaults for optimal performance

### 5. **Maintainability**
- Easy to add new permissions
- Easy to add new UI components
- Easy to refactor shared hooks without touching feature
- Clear pattern for other developers

### 5. **Follows Project Rules**
- Adheres to Feature Module Guidelines (project-rules.md)
- Implements Security First principle
- Uses Permission-Based Access control
- Backend-First architecture (thin frontend)

## 🎓 Key Takeaways

1. **Feature Modules = UI Presentation Only**
   - Components, forms, tables, loading states
   - Display formatters (formatDate, formatStatus)
   - UI-specific constants (dropdown options)

2. **Shared Code = Data & Business Logic**
   - Hooks for data fetching (`src/hooks/d1/`)
   - Domain types (`src/types/`)
   - Business logic utilities (`src/utilities/`)
   - Shared configuration (`src/constants/`)

3. **Security = Permission Guards Everywhere**
   - Every page checks permissions
   - Users see "Access Denied" for unauthorized actions
   - Backend enforces, frontend provides UX

4. **Import Directly = No Thin Wrappers**
   - Don't re-export shared code in features
   - Import hooks from `src/hooks/`
   - Import types from `src/types/`
   - Only re-export what adds value

   - No empty folders to confuse developers
5. **Clean Paths = Better Maintainability**
   - No deep relative imports in pages
   - Easy to find where code lives
   - Refactoring shared code doesn't break features
