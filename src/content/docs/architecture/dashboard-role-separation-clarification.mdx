# Dashboard Architecture - Role Separation Clarification

## ⚠️ CRITICAL: Role Isolation is PRESERVED

### Current System Roles

**From `src/types/users.ts`:**
```typescript
export type UserRole = 'superuser' | 'admin' | 'librarian' | 'dean' | 'professor' | 'student' | 'guest';
```

### Dashboard Architecture Overview

#### **Category 1: Administrative Dashboards (System-Wide Stats)**
These show aggregated system statistics and are role-specific:

| Role | Dashboard Page | Hook | Endpoint | Data Type |
|------|---------------|------|----------|-----------|
| **Superuser** | `/dashboard-superuser` | `useSuperuserDashboard` | `/api/dashboard-stats/superuser` | System health, all user stats, audit logs |
| **Admin** | `/dashboard-admin` | `useAdminDashboard` | `/api/dashboard-stats/admin` | User management stats, role distribution |
| **Librarian** | `/dashboard-librarian` | `useLibrarianDashboard` | `/api/dashboard-stats/librarian` | Book inventory, loans, overdue items |
| **Dean** | `/dashboard-dean` | `useDeanDashboard` | `/api/dashboard-stats/dean` | Faculty stats, academic overview |

**Access Pattern**: 
- Admin can also access librarian + dean dashboards
- Superuser can access ALL administrative dashboards
- Backend validates permissions on EVERY request

---

#### **Category 2: User Dashboard (Personal Stats)**
This shows personal user data, NOT system-wide stats:

| Roles | Dashboard Page | Hooks | Endpoints | Data Type |
|-------|---------------|-------|-----------|-----------|
| **Professor<br />Student<br />Guest** | `/dashboard-user` | `useUserStats`<br />`useCurrentLoans`<br />`useRecommendations`<br />`useRecentActivities` | `/api/users/{userId}/stats`<br />`/api/loans?userId={id}`<br />`/api/recommendations?userId={id}`<br />etc. | **Personal** user data only |

**Access Pattern**: 
- These are USER-SPECIFIC endpoints (require userId parameter)
- Each user ONLY sees their own data
- NOT role-based aggregated stats

---

## 🚨 What My Proposal DOES NOT DO

### ❌ **WILL NOT Mix Roles**
- Each role keeps its DEDICATED endpoint
- Each role keeps its DEDICATED hook  
- Each role keeps its DEDICATED data model
- NO cross-contamination between roles

### ❌ **WILL NOT Create United Endpoint**
- I am **NOT** proposing a single `/api/dashboard` for all roles
- Each role-specific endpoint remains independent
- Path separation preserved: `/admin`, `/librarian`, `/dean`, `/superuser`

### ❌ **WILL NOT Change Authorization**
- Backend still validates user role on EVERY request
- Permission checks stay exactly as they are
- Superuser/admin fallback permissions preserved

// useBaseDashboard.ts (Shared implementation)
export const useBaseDashboard = <TStats, TData>(config: BaseDashboardConfig<TStats, TData>) => {
  const query = useQuery({
    queryKey: [config.queryKey],
    queryFn: async () => {
      const response = await api.get<DashboardAPIResponse<TData>>(
        config.endpoint  // ← Each role passes its OWN endpoint
      );
      return config.transformResponse(response.data);
    },
    staleTime: 0,  // Consistent defaults
    refetchOnWindowFocus: true,
  });
  
  return {
    stats: query.data?.stats ?? config.defaultStats,
    data: query.data,
    loading: query.isLoading,
    error: query.error,
    refetch: query.refetch,
    dataUpdatedAt: query.dataUpdatedAt,
  };
};

## 🔍 Analogy: Shared Base Class

Think of `useBaseDashboard` like a base class in OOP:

```typescript
// Base class (shared implementation)
class BaseDashboard<TStats> {
  constructor(endpoint: string, defaultStats: TStats) {...}
  fetch() {...}
  refetch() {...}
}

// Each role has its own instance (separate data)
class AdminDashboard extends BaseDashboard<AdminStats> {
  constructor() {
    super('/api/v1/dashboard-stats/admin', DEFAULT_ADMIN_STATS);
  }
}

class LibrarianDashboard extends BaseDashboard<LibrarianStats> {
  constructor() {
    super('/api/v1/dashboard-stats/librarian', DEFAULT_LIBRARIAN_STATS);
  }
}

// Each instance is completely isolated
const adminDash = new AdminDashboard();  // ← Calls /admin only
const librarianDash = new LibrarianDashboard();  // ← Calls /librarian only
// No data mixing!
```

**Your dashboards work the same way:**
- Shared implementation (useBaseDashboard)
- Separate instances (each hook)
- Separate endpoints (each role)
- **Zero data mixing**
